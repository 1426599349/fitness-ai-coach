# 健身AI个性化健康助手

> **一个面向普通健身人群的 AI 健康助手小程序，采用 Skill + MCP + Harness 三层企业级 AI 架构。**

🔗 [在线体验](#) | 📱 微信小程序搜索「健身AI助手」

---

## 🎯 产品定位

市面上健身产品普遍存在：方案固化、AI 自由输出无约束、饮食与健身脱节。本项目通过 **AI 可控架构 + 饮食健身联动 + 双模运行机制**，实现安全、可迭代、可商业化的垂直领域 AI Agent。

### 目标用户
- 普通健身 / 减脂 / 塑形人群
- 无专业健身知识、不喜欢专业健身餐
- 想要简单、安全、个性化指导的用户

## 🏗 架构设计

```
┌─────────────────────────────────────┐
│           Harness 约束管控层          │
│   医疗红线 · 强度上限 · 忌口黑名单      │
│   主食去重 · 数据校验 · 输出格式化      │
├─────────────────────────────────────┤
│            MCP 中控调度层             │
│   流程编排 · 意图识别 · 对话记忆       │
│   定时任务 · Skill 调度 · 双模切换     │
├─────────────────────────────────────┤
│            Skill 能力原子层           │
│   数据录入 · 指标测算 · 方案生成       │
│   回访问询 · 主食随机 · 饮食搭配       │
│   偏好过滤                           │
├─────────────────────────────────────┤
│   DeepSeek API  │  微信云开发  │  本地规则引擎   │
└─────────────────────────────────────┘
```

**核心设计原则**：能力可复用、流程可管控、输出可约束、数据可迭代。

### 三层职责

| 层 | 职责 | 判定规则 |
|---|---|---|
| **Skill** | 不可拆分的最小业务能力，单一职责、独立运行 | 凡是需要复用、单独管控、单独迭代的能力 |
| **MCP** | 流程编排、判断、调度、记忆，不做具体业务计算 | 凡是流程、判断、记忆、调度、定时 |
| **Harness** | 包裹 MCP 与 Skill 的外部约束系统，代码硬约束 | 凡是安全边界、合规红线、质量校验 |

## 🔄 核心流程

### 新用户初始化
```
录入身体数据 → Harness 校验合法性 → BMI/BMR/TDEE 测算
→ AI 生成 7 天训练计划 → Harness 校验强度和安全性
→ 随机主食 + 饮食搭配（过滤忌口、优先喜欢食材）
→ 结构化输出 + 同步到本周计划页
```

### 7 天滚动计划（非固定周一）
```
计划从任意一天开始 → Day1~Day7 + 实际日期
→ AI 仅从 153 个核心动作库中选取（禁止编造）
→ 用户可勾选完成、标注难度、编辑参数
→ 再打开从本地缓存立即展示 → 后台静默刷新
```

### 训练计划 AI 自主更新
```
对话记忆（最近20条）→ 加载当前计划 + 用户档案
→ AI 自主判断是否需更新（不依赖关键词匹配）
→ 更新后自动覆盖旧计划 → 前端同步拉取
```

### 饮食推荐闭环
```
每日随机主食（去重昨日）→ 按热量目标 33%-50% 搭配
→ 自动过滤忌口食材 → 优先围绕用户喜欢的食材
→ 一键换餐（排除当日+昨日主食 + 重新搭配）
```

## 🛡 安全设计（Harness 约束层）

| 约束类型 | 实现方式 | 示例 |
|---|---|---|
| **硬规则拦截** | 代码直接拦截 | 身高 50-250cm、体重 20-300kg 区间；医疗关键词驳回 |
| **流程枷锁** | 强制前序条件 | 必须录入身体数据后才可生成方案 |
| **校验闭环** | 输出后二次校验 | 训练方案必须匹配体能等级，不合格自动重生成 |
| **输出约束** | 固定结构化输出 | AI 禁止自由闲聊跑偏；动作仅从库中选取 |

| | 云端模式 | 本地模式 |

| **对话记忆** | 云数据库 + 本地同步 | 本地 Storage |

## 🧠 产品思考

### 为什么不做自由对话
多数 AI 健身产品让大模型自由输出，存在幻觉、强度超标、饮食不合理等风险。本项目用 Harness 硬约束 + 动作库注入 + 结构化输出，确保 AI 的每一步都在可控范围内。

### 为什么本地本地兜底
健身场景下用户可能在健身房（网络差），本地规则引擎保证核心体验不中断。双模切换对用户透明。

### 为什么动作库要精选到 153 个
791 个动作中大量是冷门变体，普通用户不认识也用不上。白名单精选后，AI 只能从这里选，确保每个动作都有详细的图文说明。

### 为什么计划是 Day1~Day7 而不是周一/周二
用户可能周三才开始健身计划，"Day1" 比"周一"更符合实际使用场景。
