健身AI个性化健康助手 PRD（V3.1｜Agent + Harness + Tool Calling + 用户风控）

================================================================================
1. 文档信息
================================================================================
  文档名称：健身AI个性化健康助手 V3.1 PRD（含交互+架构说明）
  产品形态：微信小程序
  底层架构：Agent（LLM Tool Calling）+ Harness（约束管控）+ Skill（能力原子）
  产品定位：轻量化、动态迭代、高安全性的个人健身饮食AI助手
  目标用户：普通健身人群、减脂人群、无专业健身知识、需要饮食搭配指导的用户


================================================================================
2. 需求背景（简要）
================================================================================
  市面上健身类产品普遍存在：方案固化、无法随体重动态更新、饮食搭配死板、
  AI输出不专业、容易生成危险训练动作、无个性化忌口适配等问题。

  本产品基于 Agent + Harness + Skill 三层架构：
  Agent 自主决策 → 工具调用 → Harness 兜底 → Skill 执行。
  LLM 原生 Tool Calling 替代硬编码意图识别，代码量减少 79%。


================================================================================
3. 产品整体架构
================================================================================
3.1 三层架构
  Agent 层（LLM 自主决策）：对话记忆管理、意图理解、工具调用编排
    基于 DeepSeek 原生 Tool Calling，LLM 自己决定何时调用哪个工具
    替代了 V2.0 的硬编码正则意图识别 + 7 个分散 handler
    支持多轮工具调用串行/并发，最大 5 轮安全上限

  Harness 层（约束管控）：硬规则拦截、流程枷锁、数据校验、医疗红线
    前置校验（医疗红线 + 积分）在 Agent Loop 之外，不受 AI 行为影响
    后置校验（动作合法性、禁止动作、强度上限）在工具执行器中触发

  Skill 层（能力原子）：5 个可执行工具，被 Agent 通过 Tool Calling 调度
    ├── update_body_data     — 从自然语言提取并保存身体数据
    ├── generate_weekly_plan  — 生成/重新生成 7 天训练计划
    ├── generate_meal         — 生成一餐饮食方案
    ├── record_weight         — 记录体重 + 变化判定
    └── get_exercise_info     — 查询动作库

  架构进化（V2.0 → V3.0）：
    V2.0：detectIntent() 正则匹配 → switch 6 种意图 → 7 个 handler 各写各的流程
    V3.0：Agent Loop 加载上下文 → LLM 看见 5 个工具 → 自主选工具+传参 → 执行 → 回复
    删除代码 ~1000 行，意图识别代码归零

3.2 云端/本地双模运行
  云函数可用时  → 调用大模型（DeepSeek）生成个性化方案，数据库持久化
  云函数不可用时 → 本地规则引擎兜底，Storage持久化，保证核心流程不中断

3.3 对话记忆系统
  每次对话后将"用户消息+AI回复"存入 user_states.conversationHistory
  保留最近 20 条消息，以原始消息数组格式直接注入 Agent 的 messages 列表
  Agent 通过 system prompt + 历史消息 + 用户最新消息，决策是否调用工具
  云端不可用时，本地 Storage 同样维护 conversationHistory

4. 页面结构
================================================================================
  小程序共 5 个页面：

  ┌──────────────────────────────────────────────────┐
  │ TabBar                                              │
  │  [AI教练]  [饮食转盘]  [我的]                        │
  └──────────────────────────────────────────────────┘

  4.1 AI教练（pages/chat）—— 对话首页
      核心对话界面，用户在此：
      - 首次录入身体数据（身高/体重/年龄/性别/目标/场所）
      - 与AI自由对话，获取训练建议、饮食建议
      - 查看AI生成的训练计划摘要
      - 更新体重、调整训练计划

  4.2 饮食转盘（pages/wheel）—— 每日三餐
      每日随机生成早/午/晚三餐，三顿主食不重复
      基于用户身体数据个性化匹配热量
      支持重新转（消耗积分或看广告）

  4.3 我的（pages/profile）—— 个人中心
      查看/编辑身体档案
      查看体重历史
      编辑饮食忌口
      入口：本周训练计划、动作库

  4.4 食物库（pages/food-lib）—— Tab 页
      浏览 160 种食物，按主食/主菜/配菜三个分类筛选
      每项显示名称、每100g热量、蛋白质/脂肪/碳水
      点击进入详情页（nutrition detail card + 热量来源柱状图）

  4.5 本周训练计划（pages/weekly-plan）—— 从"我的"进入
      展示 AI 生成的完整 7 天训练计划
      每天训练/休息、具体动作、组数/次数/重量
      支持标注难度（简单/适中/困难）、手动编辑参数

  4.5 动作库（pages/exercise-lib + exercise-detail）—— 从"我的"进入
      按肌群分类浏览 900+ 训练动作
      查看动作详情

5. 核心交互流程（详细）
================================================================================

========================================
5.1 新用户初始化流程
========================================

  触发条件：用户首次打开AI教练页，或用户档案中无身体数据

  步骤1：AI 发送欢迎语 + 引导语
    文案：「你好，我是你的智能营养管家 🥗
          请填写以下信息，我会为你定制专属计划」
    同时展示录入表单（身高/体重/年龄/性别/目标）

  步骤2：用户输入身体数据
    支持两种方式：
    a) 自然语言：如"170 75 男 28 减脂 居家"
    b) 表单提交：逐个填写字段后点击提交

  步骤3：Harness 前置校验
    身高 50-250cm / 体重 20-300kg / 年龄 10-120
    性别仅限 male/female
    健身目标仅限 fat_loss/muscle_gain/shape/maintain
    体能等级仅限 beginner/intermediate/advanced
    不合法数据 → 提示错误原因，拒绝录入

  步骤4：Agent 自主决定调用工具链
    Agent 识别到用户提供了身体数据 → 调用 update_body_data 工具
      工具内部：Harness 校验 → 保存到数据库 → 计算 BMI/BMR/TDEE → 返回结果
    Agent 看到数据完整 + 场所已选 → 调用 generate_weekly_plan 工具
      工具内部：拼动作库 prompt → DeepSeek 生成 7 天计划 → Harness 校验 → 存库 → 返回摘要
    每步工具结果返回给 Agent，Agent 据此决定下一步

  步骤5：Harness 后置校验
    训练方案：动作合法性、强度合规、高危动作拦截
    饮食方案：忌口过滤、热量区间校验、主食去重

  步骤6：结构化输出
    AI 回复文案包含：
    - BMI、每日消耗热量
    - 7天计划摘要（周一到周日训练分布）
    - 引导用户去「本周训练计划」查看详情

  步骤7：计划同步
    前端自动调用 get_weekly_plan 拉取最新计划
    写入本地 Storage（key: weeklyPlan）
    本周训练计划页 onShow 时立即渲染

  步骤8：状态转换
    对话状态 convState: greeting → onboarding → ready
    ready 状态后展示快捷操作按钮（今天吃什么/换一换/训练计划/更新体重）

========================================
5.2 每日饮食生成流程
========================================

  触发方式：
    a) 用户说"今天吃什么""饮食""餐单"
    b) 点击快捷按钮「今天吃什么」
    c) 每日定时任务自动触发（云端定时）

  流程：
    Agent 识别用户要饮食推荐 → 调用 generate_meal 工具（无 regenerate 参数） →
    工具内部：加载用户档案（忌口/喜欢食材） →
    调用 DeepSeek AI 生成单餐（或规则引擎兜底） →
    Harness 校验热量区间（目标热量的 33%-50%）+ 忌口过滤 →
    返回结构化饮食卡片（main_staple / dishes[] / total_kcal / 营养配比）→
    Agent 看到结果后组织友好回复

  前端展示：
    饮食卡片（meal-card 组件），显示主食名+克数、各道菜名+克数+热量、总热量和营养

========================================
5.3 重新生成饮食流程
========================================

  触发方式：
    a) 用户说"换一换""换餐""换个""再来一份"
    b) 点击快捷按钮「换一换」
    c) 点击饮食卡片上的「换一换」按钮

  流程：
    Agent 识别用户要换餐 → 调用 generate_meal({ regenerate: true }) →
    工具内部：加载当日已记录主食 + 昨日主食 →
    重新随机主食（排除当日+昨日） → 重新搭配饮食 →
    Harness 校验（与 5.2 相同） →
    返回新饮食卡片

  注意：
    每餐热量控制在目标热量的 33%-50%
    禁用品出现在忌口列表中的食材
    连续两次生成同款主食会被 Harness 强制去重

========================================
5.4 体重更新流程
========================================

  触发方式：
    a) 用户说"更新体重""体重72kg""称重"
    b) 输入格式：数字 + 可选单位（kg/公斤/斤）

  流程：
    Agent 识别用户报体重 → 调用 record_weight({ weight, unit }) →
    工具内部：单位转换（斤→kg）→ 加载历史体重，计算变化百分比 →
    保存体重记录（weight_logs 集合）→ 更新用户档案中的体重字段 →

    工具返回结果：
      变化 ≤ 2% → { shouldRegenerate: false, message: "体重变化平稳" }
      变化 > 2% → { shouldRegenerate: true, message: "体重变化 X%，建议更新计划" }

    Agent 看到 shouldRegenerate=true → 调用 generate_weekly_plan →
    计划同步：planUpdated=true → 前端拉取新计划 → 写入 Storage

========================================
5.5 训练计划调整流程（AI自主判断）
========================================

  触发方式：用户在对话中表达调整意图
    示例："太简单了""太难了""换动作""加强腿部""减轻强度""帮我调整计划"

  流程：
    Agent Loop 加载上下文（用户档案 + 当前计划 + 对话记忆） →
    将上下文注入 system prompt + 对话历史注入 messages →

    Agent 自主判断逻辑（LLM 原生推理，无需代码显式判断）：
    - 用户闲聊/咨询/问建议 → 不调工具，直接文字回复
    - 用户表达改变/调整/修改/替换/加强/减轻意图 →
      调用 generate_weekly_plan({ feedback: "用户的调整诉求" })
    - 结合对话记忆理解连续意图（如用户先说了"居家"，再说"太简单"，
      Agent 知道是居家计划太简单，feedback 参数自然包含此上下文）

    工具执行 → Harness 校验 → 保存到 user_states.weeklyPlan（覆盖旧计划） →
    设置 planUpdated: true → Agent 收到结果后组织回复 →

    前端收到响应 → syncWeeklyPlan() → 拉取最新计划 → 写入 Storage →
    本周训练计划页自动展示新版本

========================================
5.6 本周训练计划页交互
========================================

  页面路径：我的 → 本周训练计划（pages/weekly-plan）

  加载逻辑（onShow 每次触发）：
    第1层：loadFromCache()
      从 wx.Storage 读取 weeklyPlan
      有数据 → 立即渲染（0ms响应）
      无数据 → 显示 loading

    第2层：loadFromCloud()（后台静默）
      调用云函数 get_weekly_plan 获取最新数据
      成功 → 覆盖本地缓存 + 刷新页面
      失败 → 保留本地缓存展示（不清空）

  页面展示：
    - 顶部标题「训练计划 本周」
    - 7张卡片（周一到周日）：
      - 休息日：星期标签 + "休息日 💤"
      - 训练日：星期标签 + 训练部位 + 动作列表
        - 每个动作显示：名称 / 组数 / 次数 / 重量 / 注意事项
        - 难度标注按钮：简单 ✅ / 适中 ✅ / 困难 ✅

  用户可操作：
    a) 标注难度：点击"简单/适中/困难"按钮
        → 保存到 planEdits Storage（按动作名称索引）
        → 标注后参数颜色变化

    b) 编辑参数：点击组数/次数/重量数字
        → 弹出输入框
        → 用户输入新值（范围 1-99）
        → 保存到 planEdits + 更新展示

  空状态：
    无计划时展示"暂无训练计划" + 引导文案
    "去「AI教练」录入身体数据后，计划会自动生成"

========================================
5.7 对话记忆详细逻辑
========================================

  存储位置：
    云端：user_states.conversationHistory（字段，数组类型）
    本地：wx.Storage.conversationHistory

  存储格式：
    [
      { role: "user", content: "170 75 男 28 减脂" },
      { role: "assistant", content: "已记录并生成本周计划..." },
      { role: "user", content: "太简单了" },
      { role: "assistant", content: "了解，帮你加强了训练强度..." },
      ...
    ]

  保留策略：最近 20 条（MAX_HISTORY = 20）

  喂给AI的方式：
    在 Agent Loop 中，对话历史以标准 messages 数组格式直接注入：
    [
      { role: "system", content: systemPrompt },
      { role: "user", content: "170 75 男 28 减脂" },
      { role: "assistant", content: "已记录并生成本周计划..." },
      { role: "user", content: "太简单了" },
      ...
    ]
    Agent 以此完整上下文理解用户连续意图，无需手动拼接格式化文本。

  触发时机：
    Agent Loop 结束后，自动调用 saveConversationTurn 保存本轮对话
    仅在对话消息（非 action 查询类）时保存

========================================
5.8 双模运行机制
========================================

  云函数可用时（正常模式）：
    api.aiChat()
      → callCloudFunction('aiChat', { message })
      → index.js: Harness 前置（红线+积分） → Agent Loop
      → Agent Loop: 加载上下文 → system prompt → 对话历史 →
        循环（最多5轮）：callAI(messages, tools) → 有 tool_calls 就执行 → 无则 break
      → 保存对话记忆 → 返回 { type, content, credits, planUpdated }
      → renderAIResponse() 处理展示
      → syncWeeklyPlan() 主动拉取最新计划

  云函数不可用时（本地兜底）：
    api.aiChat()
      → callCloudFunction 调用失败（超时/未部署/未初始化）
      → localAiResponse() 本地规则引擎接管（V2.0 逻辑不变）
      → 对话记忆存储在 wx.Storage
      → 计划存储在 wx.Storage

  本地引擎覆盖场景：
    - 身体数据录入（解析自然语言 → 生成计划）
    - 计划修改意图检测（太简单/太难/换动作等关键词）
    - 体重更新（计算变化百分比 → 决定是否更新计划）
    - 饮食查询（随机本地菜单）
    - 训练计划查询（读取 Storage）
    - 通用回复（引导用户使用可用功能）

  切换对用户透明：前端行为一致，不弹出技术性错误提示

6. 产品功能详细需求
================================================================================

6.1 用户身体档案模块
  ├── 基础信息：身高(cm)、体重(kg)、年龄、性别
  ├── 运动目标：减脂/增肌/塑形/维持身材
  ├── 运动基础：零基础/初级/中级
  ├── 运动场所：居家/健身房
  ├── 饮食忌口：手动添加忌口食材列表
  ├── 每周体重记录、历史展示
  └── 数据校验：身高50-250cm、体重20-300kg、年龄10-120

6.2 AI健身方案模块
  ├── 初次生成：录入数据后 Agent 自主调用 generate_weekly_plan 工具
  ├── 训练分配：Day1 胸+三头、Day2 背+二头、Day3 休息、
  │            Day4 腿+肩、Day5 核心+有氧、Day6/7 休息（非固定周一，滚动 7 天）
  ├── 每周迭代：体重变化 >2% → record_weight 返回 shouldRegenerate=true →
  │            Agent 接着调 generate_weekly_plan 自动重生成
  ├── Agent 自主调整：用户表达调整意图 → LLM 自行决定调用
  │            generate_weekly_plan({ feedback }) 工具
  ├── 动作详情：名称、组数、次数、重量(kg)、休息时间、注意事项
  ├── 难度分级：初级/中级/高级，适配不同人群
  ├── 场所适配：居家（自重/哑铃）vs 健身房（器械/自由重量）
  ├── 禁止动作：颈后推举、颈后下拉、早安式体前屈、
  │            断头台卧推、直立划船等（Harness 黑名单硬拦截）
  └── 强度管控：按体能等级限制动作数/组数/次数/训练时长

6.3 每日智能饮食模块
  ├── 主食随机：从23种主食中随机选取（白米饭/糙米饭/面条/
  │            饺子/红薯/全麦面包/玉米等）
  ├── 配菜搭配：1道主菜（蛋白质）+ 1-2道配菜（蔬菜）
  ├── 热量控制：每餐热量 = 目标热量的33%-50%
  ├── 营养展示：蛋白质(g)、脂肪(g)、碳水(g)、总热量(kcal)
  ├── 重新生成：一键换餐，排除当前和昨日主食
  ├── 忌口过滤：自动排除用户忌口食材
  └── 生活化：家常菜为主，拒绝专业健身餐

6.4 对话记忆模块（Agent 层，用户无感知）
  ├── 存储位置：user_states.conversationHistory（云端）+ wx.Storage（本地）
  ├── 保留数量：最近 20 条消息
  ├── 每次 Agent Loop 结束后自动保存：用户消息 + AI 回复
  ├── 下次对话时以标准 messages 数组格式注入 Agent 上下文
  └── 无需手动拼接格式化文本，LLM 原生理解对话角色

6.5 Harness管控中心（纯后台）
  ├── 数据合法性校验（身高/体重/年龄区间）
  ├── 医疗红线拦截（治疗/诊断/处方/手术等关键词）
  ├── 训练强度阈值管控（按体能等级）
  ├── 高危动作黑名单过滤
  ├── 饮食黑名单过滤（忌口食材）
  ├── 主食连续去重（禁止连续两天同款）
  ├── 输出格式约束（结构化JSON，禁止自由闲聊跑偏）
  ├── 深夜休息时段风控（23:00-07:00，AI能力受限）    ← V3.1 新增
  └── 全链路日志留存

6.6 用户身份模块（V3.1 新增）
  ├── 入口：个人中心 → 点头像选头像 + 点击昵称编辑
  ├── 方案：<button open-type="chooseAvatar"> + <input type="nickname">
  │        （微信官方推荐方案，wx.getUserProfile 已于 2023 年废弃）
  ├── 合规性：个人开发者完全可用，无需额外权限申请

  └── 默认身份：未设置时显示 emoji 🏋️ + "健身爱好者"

8. 计划同步机制（核心）
================================================================================

  8.1 同步触发时机（chat.js renderAIResponse）
      每次 AI 回复后，无论回复类型（text/metrics/workout/onboarding/meal），
      均调用 syncWeeklyPlan() 主动从云端拉取最新计划。

  8.2 同步数据流
      AI 生成/更新计划 → 云数据库 user_states.weeklyPlan（写入）
      ↓
      前端 syncWeeklyPlan() → 调用云函数 get_weekly_plan（读取）
      ↓
      wx.setStorageSync('weeklyPlan', plan)  （本地缓存）
      ↓
      本周计划页 onShow → loadFromCache()（立即展示）
                       → loadFromCloud()（后台刷新确认）

  8.3 计划覆盖规则
      - 新用户初始化：首次生成 → 写入
      - 每周体重回访（变化>2%）：重新生成 → 覆盖
      - AI自主判断（用户表达调整意图）：重新生成 → 覆盖
      - 用户手动要求：重新生成 → 覆盖
      - 每次覆盖都是全量替换（整个 weeklyPlan 对象），不存在增量更新

  8.4 本地存储 key 清单
      weeklyPlan          — 当前7天训练计划（JSON，与云端同格式）
      planEdits           — 用户手动编辑的参数（按动作名称索引）
      conversationHistory — 对话记忆（最近20条）
      userProfile         — 用户档案（app.globalData，非 Storage）

  8.5 空状态处理
      无计划时本周计划页展示引导文案，引导用户去AI教练页录入数据。

9. 异常流程规则（Harness专属）
================================================================================

  1. 用户输入离谱体重/身高：
     → 直接拦截，提示"身高需在50-250cm之间""体重需在20-300kg之间"

  2. AI生成高危动作：
     → Harness规则拦截（BANNED_EXERCISES黑名单），自动重生成方案

  3. 生成饮食包含忌口食材：
     → Harness 强制过滤，重新搭配

  4. 连续两天生成同款主食：
     → Harness 去重规则，强制排除昨日主食

  5. 用户询问医疗/伤病问题：
     → 医疗红线拦截，统一回复：
     "您好，我是健身饮食助手，不提供医疗建议。如有健康问题，请咨询专业医生。"

  6. 云函数不可用：
     → 本地规则引擎接管，对用户透明，核心功能不受影响

  7. 用户未录入身体数据时请求训练/饮食：
     → 引导录入："请先告诉我你的身体数据~"

  8. 积分不足：
     → 提示"积分不足！（当前X分），看一次视频广告可获得10积分"

  9. 对话记忆存储失败：
     → 静默处理，不影响主流程

  10. 计划同步失败：
      → 保留本地缓存，不清空已展示内容

10. 非功能需求
================================================================================

  10.1 性能
      - Agent 单轮对话耗时 ≤ 5s（含 1-2 轮 Tool Calling）
      - 多轮工具调用（如 录入数据+生成计划）≤ 12s
      - 本地方案生成耗时 ≤ 300ms
      - 饮食生成耗时 ≤ 2s
      - 本周计划页首屏渲染 ≤ 500ms（本地缓存）

  10.2 安全
      - 用户身体数据加密存储
      - API Key 使用云函数环境变量，不硬编码
      - 医疗红线关键词拦截
      - 禁止明文传输隐私数据

  10.3 兼容性
      - 适配微信小程序（基础库 ≥ 2.2.3）
      - 适配所有手机尺寸（rpx响应式布局）

  10.4 可用性
      - 云端不可用时本地引擎兜底
      - 对话记忆双重存储（云端+本地）
      - 计划数据双重存储（云端+本地）
