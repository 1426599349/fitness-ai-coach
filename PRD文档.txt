健身AI个性化健康助手 PRD（V2.0 含详细交互｜MCP+Harness+Skill架构）

================================================================================
1. 文档信息
================================================================================
  文档名称：健身AI个性化健康助手 V2.0 PRD（含详细交互说明）
  产品形态：微信小程序
  底层架构：Skill（能力原子）+ MCP（中控调度）+ Harness（约束管控）
  产品定位：轻量化、动态迭代、高安全性的个人健身饮食AI助手
  目标用户：普通健身人群、减脂人群、无专业健身知识、需要饮食搭配指导的用户


================================================================================
2. 需求背景（简要）
================================================================================
  市面上健身类产品普遍存在：方案固化、无法随体重动态更新、饮食搭配死板、
  AI输出不专业、容易生成危险训练动作、无个性化忌口适配等问题。

  本产品基于MCP+Harness+Skill三层企业级AI架构：
  能力可复用、流程可管控、输出可约束、数据可迭代。


================================================================================
3. 产品整体架构
================================================================================
3.1 三层架构
  Skill层（能力原子）：7个独立Skill，单一职责，MCP调度调用
    ├── 用户身体数据录入 Skill
    ├── 人体指标测算 Skill
    ├── 健身方案生成 Skill
    ├── 每周回访问询 Skill
    ├── 每日主食随机生成 Skill
    ├── 每日饮食搭配 Skill
    └── 用户偏好过滤 Skill

  MCP层（中控调度）：流程编排、意图识别、对话记忆、定时调度、Skill串联

  Harness层（约束管控）：硬规则拦截、流程枷锁、数据校验、医疗红线

3.2 云端/本地双模运行
  云函数可用时  → 调用大模型（DeepSeek）生成个性化方案，数据库持久化
  云函数不可用时 → 本地规则引擎兜底，Storage持久化，保证核心流程不中断

3.3 对话记忆系统
  每次对话后将"用户消息+AI回复"存入 user_states.conversationHistory
  保留最近 20 条消息，下次对话时喂给 AI 作为上下文
  AI 结合记忆自主判断用户意图（是否需要更新计划、调整饮食等）
  云端不可用时，本地 Storage 同样维护 conversationHistory

4. 页面结构
================================================================================
  小程序共 5 个页面：

  ┌──────────────────────────────────────────────────┐
  │ TabBar                                              │
  │  [AI教练]  [饮食转盘]  [我的]                        │
  └──────────────────────────────────────────────────┘

  4.1 AI教练（pages/chat）—— 对话首页
      核心对话界面，用户在此：
      - 首次录入身体数据（身高/体重/年龄/性别/目标/场所）
      - 与AI自由对话，获取训练建议、饮食建议
      - 查看AI生成的训练计划摘要
      - 更新体重、调整训练计划

  4.2 饮食转盘（pages/wheel）—— 每日三餐
      每日随机生成早/午/晚三餐，三顿主食不重复
      基于用户身体数据个性化匹配热量
      支持重新转（消耗积分或看广告）

  4.3 我的（pages/profile）—— 个人中心
      查看/编辑身体档案
      查看体重历史
      编辑饮食忌口
      入口：本周训练计划、动作库

  4.4 食物库（pages/food-lib）—— Tab 页
      浏览 160 种食物，按主食/主菜/配菜三个分类筛选
      每项显示名称、每100g热量、蛋白质/脂肪/碳水
      点击进入详情页（nutrition detail card + 热量来源柱状图）

  4.5 本周训练计划（pages/weekly-plan）—— 从"我的"进入
      展示 AI 生成的完整 7 天训练计划
      每天训练/休息、具体动作、组数/次数/重量
      支持标注难度（简单/适中/困难）、手动编辑参数

  4.5 动作库（pages/exercise-lib + exercise-detail）—— 从"我的"进入
      按肌群分类浏览 900+ 训练动作
      查看动作详情

5. 核心交互流程（详细）
================================================================================

========================================
5.1 新用户初始化流程
========================================

  触发条件：用户首次打开AI教练页，或用户档案中无身体数据

  步骤1：AI 发送欢迎语 + 引导语
    文案：「你好，我是你的智能营养管家 🥗
          请填写以下信息，我会为你定制专属计划」
    同时展示录入表单（身高/体重/年龄/性别/目标）

  步骤2：用户输入身体数据
    支持两种方式：
    a) 自然语言：如"170 75 男 28 减脂 居家"
    b) 表单提交：逐个填写字段后点击提交

  步骤3：Harness 前置校验
    身高 50-250cm / 体重 20-300kg / 年龄 10-120
    性别仅限 male/female
    健身目标仅限 fat_loss/muscle_gain/shape/maintain
    体能等级仅限 beginner/intermediate/advanced
    不合法数据 → 提示错误原因，拒绝录入

  步骤4：MCP 调度 Skill 链
    Skill1 用户身体数据录入 → 保存到数据库
    Skill2 人体指标测算 → BMI / BMR / TDEE / 建议摄入热量
    Skill3 健身方案生成 → 7天训练计划（规则引擎+AI混合）
    Skill5+6 主食随机+饮食搭配 → 当日餐单

  步骤5：Harness 后置校验
    训练方案：动作合法性、强度合规、高危动作拦截
    饮食方案：忌口过滤、热量区间校验、主食去重

  步骤6：结构化输出
    AI 回复文案包含：
    - BMI、每日消耗热量
    - 7天计划摘要（周一到周日训练分布）
    - 引导用户去「本周训练计划」查看详情

  步骤7：计划同步
    前端自动调用 get_weekly_plan 拉取最新计划
    写入本地 Storage（key: weeklyPlan）
    本周训练计划页 onShow 时立即渲染

  步骤8：状态转换
    对话状态 convState: greeting → onboarding → ready
    ready 状态后展示快捷操作按钮（今天吃什么/换一换/训练计划/更新体重）

========================================
5.2 每日饮食生成流程
========================================

  触发方式：
    a) 用户说"今天吃什么""饮食""餐单"
    b) 点击快捷按钮「今天吃什么」
    c) 每日定时任务自动触发（云端定时）

  流程：
    MCP 识别意图为 meal →
    加载用户档案（忌口列表） →
    调用 Skill 随机主食（排除昨日主食 + 用户忌口） →
    调用 Skill 饮食搭配（1主菜 + 1-2配菜） →
    Harness 校验热量区间（目标热量的33%-50%） →
    返回结构化饮食卡片（main_staple / dishes[] / total_kcal / 营养配比）

  前端展示：
    饮食卡片（meal-card 组件），显示主食名+克数、各道菜名+克数+热量、总热量和营养

========================================
5.3 重新生成饮食流程
========================================

  触发方式：
    a) 用户说"换一换""换餐""换个""再来一份"
    b) 点击快捷按钮「换一换」
    c) 点击饮食卡片上的「换一换」按钮

  流程：
    MCP 识别意图为 regenerate_meal →
    加载当日已记录主食 + 昨日主食 →
    Skill 随机主食（排除当日+昨日） →
    Skill 重新搭配饮食 →
    Harness 校验（与我5.2相同） →
    返回新饮食卡片

  注意：
    每餐热量控制在目标热量的 33%-50%
    禁用品出现在忌口列表中的食材
    连续两次生成同款主食会被 Harness 强制去重

========================================
5.4 体重更新流程
========================================

  触发方式：
    a) 用户说"更新体重""体重72kg""称重"
    b) 输入格式：数字 + 可选单位（kg/公斤/斤）

  流程：
    MCP 从消息中提取体重数值 →
    （如果是"斤"单位，除以2转换为kg） →
    加载历史体重，计算变化百分比 →
    保存体重记录（weight_logs 集合） →
    更新用户档案中的体重字段 →

    判断逻辑：
      变化 ≤ 2% → 回复"体重变化平稳，继续执行原计划"
      变化 > 2% → Skill 重新生成健身方案 → 同步更新本周计划
        → 回复"体重变化 X%，超过2%，已为你更新训练方案"

    计划同步：planUpdated=true → 前端拉取新计划 → 写入 Storage

========================================
5.5 训练计划调整流程（AI自主判断）
========================================

  触发方式：用户在对话中表达调整意图
    示例："太简单了""太难了""换动作""加强腿部""减轻强度""帮我调整计划"

  流程：
    MCP 加载对话记忆（最近20条） →
    加载用户档案 + 当前训练计划摘要 →
    将所有上下文喂给 AI →

    AI 自主判断逻辑（由 system prompt 约束）：
    - 用户闲聊/咨询/问建议 → 只回复，不更新计划
    - 用户表达改变/调整/修改/替换/加强/减轻意图 → 生成新的完整7天计划
    - 结合对话记忆理解连续意图（如先说了"居家"，再说"太简单"，AI知道是居家计划太简单）

    AI 返回 JSON 包含新 plan 对象 →
    MCP 保存到 user_states.weeklyPlan（覆盖旧计划）→
    返回 planUpdated: true →

    前端收到响应 → syncWeeklyPlan() → 拉取最新计划 → 写入 Storage →
    本周训练计划页自动展示新版本

========================================
5.6 本周训练计划页交互
========================================

  页面路径：我的 → 本周训练计划（pages/weekly-plan）

  加载逻辑（onShow 每次触发）：
    第1层：loadFromCache()
      从 wx.Storage 读取 weeklyPlan
      有数据 → 立即渲染（0ms响应）
      无数据 → 显示 loading

    第2层：loadFromCloud()（后台静默）
      调用云函数 get_weekly_plan 获取最新数据
      成功 → 覆盖本地缓存 + 刷新页面
      失败 → 保留本地缓存展示（不清空）

  页面展示：
    - 顶部标题「训练计划 本周」
    - 7张卡片（周一到周日）：
      - 休息日：星期标签 + "休息日 💤"
      - 训练日：星期标签 + 训练部位 + 动作列表
        - 每个动作显示：名称 / 组数 / 次数 / 重量 / 注意事项
        - 难度标注按钮：简单 ✅ / 适中 ✅ / 困难 ✅

  用户可操作：
    a) 标注难度：点击"简单/适中/困难"按钮
        → 保存到 planEdits Storage（按动作名称索引）
        → 标注后参数颜色变化

    b) 编辑参数：点击组数/次数/重量数字
        → 弹出输入框
        → 用户输入新值（范围 1-99）
        → 保存到 planEdits + 更新展示

  空状态：
    无计划时展示"暂无训练计划" + 引导文案
    "去「AI教练」录入身体数据后，计划会自动生成"

========================================
5.7 对话记忆详细逻辑
========================================

  存储位置：
    云端：user_states.conversationHistory（字段，数组类型）
    本地：wx.Storage.conversationHistory

  存储格式：
    [
      { role: "user", content: "170 75 男 28 减脂" },
      { role: "assistant", content: "已记录并生成本周计划..." },
      { role: "user", content: "太简单了" },
      { role: "assistant", content: "了解，帮你加强了训练强度..." },
      ...
    ]

  保留策略：最近 20 条（MAX_HISTORY = 20）

  喂给AI的方式：
    在 handleGeneralChat 的 system prompt 中追加：
    【对话记忆】以下是本次会话中用户之前说的话和你的回复，
    请结合记忆理解用户的连续意图：
    用户：170 75 男 28 减脂
    AI：已记录并生成本周计划...
    用户：太简单了
    ...

  触发时机：
    每次 exports.main 处理完用户消息后，自动调用 saveConversationTurn
    仅在非查询类 action（event.action 为空）时保存

========================================
5.8 双模运行机制
========================================

  云函数可用时（正常模式）：
    api.aiChat()
      → callCloudFunction('aiChat', { message, context })
      → 云函数处理 → 调用 DeepSeek API → 返回结果
      → renderAIResponse() 处理展示
      → syncWeeklyPlan() 主动拉取最新计划

  云函数不可用时（本地兜底）：
    api.aiChat()
      → callCloudFunction 调用失败（超时/未部署/未初始化）
      → localAiResponse() 本地规则引擎接管
      → 对话记忆存储在 wx.Storage
      → 计划存储在 wx.Storage

  本地引擎覆盖场景：
    - 身体数据录入（解析自然语言 → 生成计划）
    - 计划修改意图检测（太简单/太难/换动作等关键词）
    - 体重更新（计算变化百分比 → 决定是否更新计划）
    - 饮食查询（随机本地菜单）
    - 训练计划查询（读取 Storage）
    - 通用回复（引导用户使用可用功能）

  切换对用户透明：前端行为一致，不弹出技术性错误提示

6. 产品功能详细需求
================================================================================

6.1 用户身体档案模块
  ├── 基础信息：身高(cm)、体重(kg)、年龄、性别
  ├── 运动目标：减脂/增肌/塑形/维持身材
  ├── 运动基础：零基础/初级/中级
  ├── 运动场所：居家/健身房
  ├── 饮食忌口：手动添加忌口食材列表
  ├── 每周体重记录、历史展示
  └── 数据校验：身高50-250cm、体重20-300kg、年龄10-120

6.2 AI健身方案模块
  ├── 初次生成：根据用户档案生成专属7天训练计划
  ├── 训练分配：周一胸+三头、周二背+二头、周三休息、
  │            周四腿+肩、周五核心+有氧、周六休息、周日休息
  ├── 每周迭代：体重变化>2%自动重新生成
  ├── AI自主调整：用户表达调整意图时AI自行判断是否需要更新
  ├── 动作详情：名称、组数、次数、重量(kg)、休息时间、注意事项
  ├── 难度分级：零基础/初级/中级/高级，适配不同人群
  ├── 场所适配：居家（自重/哑铃）vs 健身房（器械/自由重量）
  ├── 禁止动作：颈后推举、颈后下拉、早安式体前屈、
  │            断头台卧推、直立划船等
  └── 强度管控：按体能等级限制动作数/组数/次数/训练时长

6.3 每日智能饮食模块
  ├── 主食随机：从23种主食中随机选取（白米饭/糙米饭/面条/
  │            饺子/红薯/全麦面包/玉米等）
  ├── 配菜搭配：1道主菜（蛋白质）+ 1-2道配菜（蔬菜）
  ├── 热量控制：每餐热量 = 目标热量的33%-50%
  ├── 营养展示：蛋白质(g)、脂肪(g)、碳水(g)、总热量(kcal)
  ├── 重新生成：一键换餐，排除当前和昨日主食
  ├── 忌口过滤：自动排除用户忌口食材
  └── 生活化：家常菜为主，拒绝专业健身餐

6.4 对话记忆模块（MCP层，用户无感知）
  ├── 存储位置：user_states.conversationHistory
  ├── 保留数量：最近20条消息
  ├── 每次对话后自动保存用户消息+AI回复
  ├── 下次对话时加载并喂给AI作为上下文
  └── 本地Storage同步备份

6.5 Harness管控中心（纯后台）
  ├── 数据合法性校验（身高/体重/年龄区间）
  ├── 医疗红线拦截（治疗/诊断/处方/手术等关键词）
  ├── 训练强度阈值管控（按体能等级）
  ├── 高危动作黑名单过滤
  ├── 饮食黑名单过滤（忌口食材）
  ├── 主食连续去重（禁止连续两天同款）
  ├── 输出格式约束（结构化JSON，禁止自由闲聊跑偏）
  └── 全链路日志留存

7. 页面交互细节

========================================
7.1 AI教练页（chat）
========================================

  顶部：
    标题"AI教练"
    右侧积分显示（💎 XX 分）

  对话区域：
    - 消息气泡（用户右侧/AI左侧）
    - AI消息类型：
      · 文本消息（chat-bubble）
      · 身体指标卡片（metrics-card）：BMI/BMR/TDEE/建议摄入/营养配比
      · 训练计划卡片（workout-card）：场所/天数/每日动作
      · 饮食卡片（meal-card）：主食/配菜/热量/营养
      · 引导表单（onboarding-form）：首次录入身体数据
      · 场所选择卡片（place-choice）：居家 还是 健身房

  底部输入区：
    - 文本输入框
    - 发送按钮
    - 快捷操作栏（ready状态后展示）：
      [🍽️ 今天吃什么] [🔄 换一换] [💪 训练计划] [⚖️ 更新体重]

  积分系统：
    - 默认 200 积分
    - 每次提问消耗 5 积分
    - 首次录入身体数据不扣分
    - 看广告 +10 积分（开发阶段弹窗模拟）
    - 积分不足时提示看广告

========================================
7.2 饮食转盘页（wheel）
========================================

  顶部标题"今日三餐"

  三列转盘结果：早餐 / 午餐 / 晚餐
    每列显示：主食名称、配菜列表

  底部「重新转」按钮
    消耗 10 积分或看广告

  点击某餐可展开详情：每道菜的克数、热量

  未录入数据时：提示"请先在AI教练录入身体数据"

========================================
7.3 个人中心页（profile）
========================================

  顶部用户信息：
    头像（默认头像）
    昵称

  身体数据卡片：
    身高 / 体重 / BMI / 目标 / 每日建议热量

  体重历史：
    最近14条体重记录列表，显示变化趋势（↑/↓）

  功能入口：

8. 计划同步机制（核心）
================================================================================

  8.1 同步触发时机（chat.js renderAIResponse）
      每次 AI 回复后，无论回复类型（text/metrics/workout/onboarding/meal），
      均调用 syncWeeklyPlan() 主动从云端拉取最新计划。

  8.2 同步数据流
      AI 生成/更新计划 → 云数据库 user_states.weeklyPlan（写入）
      ↓
      前端 syncWeeklyPlan() → 调用云函数 get_weekly_plan（读取）
      ↓
      wx.setStorageSync('weeklyPlan', plan)  （本地缓存）
      ↓
      本周计划页 onShow → loadFromCache()（立即展示）
                       → loadFromCloud()（后台刷新确认）

  8.3 计划覆盖规则
      - 新用户初始化：首次生成 → 写入
      - 每周体重回访（变化>2%）：重新生成 → 覆盖
      - AI自主判断（用户表达调整意图）：重新生成 → 覆盖
      - 用户手动要求：重新生成 → 覆盖
      - 每次覆盖都是全量替换（整个 weeklyPlan 对象），不存在增量更新

  8.4 本地存储 key 清单
      weeklyPlan          — 当前7天训练计划（JSON，与云端同格式）
      planEdits           — 用户手动编辑的参数（按动作名称索引）
      conversationHistory — 对话记忆（最近20条）
      userProfile         — 用户档案（app.globalData，非 Storage）

  8.5 空状态处理
      无计划时本周计划页展示引导文案，引导用户去AI教练页录入数据。

9. 异常流程规则（Harness专属）
================================================================================

  1. 用户输入离谱体重/身高：
     → 直接拦截，提示"身高需在50-250cm之间""体重需在20-300kg之间"

  2. AI生成高危动作：
     → Harness规则拦截（BANNED_EXERCISES黑名单），自动重生成方案

  3. 生成饮食包含忌口食材：
     → Harness 强制过滤，重新搭配

  4. 连续两天生成同款主食：
     → Harness 去重规则，强制排除昨日主食

  5. 用户询问医疗/伤病问题：
     → 医疗红线拦截，统一回复：
     "您好，我是健身饮食助手，不提供医疗建议。如有健康问题，请咨询专业医生。"

  6. 云函数不可用：
     → 本地规则引擎接管，对用户透明，核心功能不受影响

  7. 用户未录入身体数据时请求训练/饮食：
     → 引导录入："请先告诉我你的身体数据~"

  8. 积分不足：
     → 提示"积分不足！（当前X分），看一次视频广告可获得10积分"

  9. 对话记忆存储失败：
     → 静默处理，不影响主流程

  10. 计划同步失败：
      → 保留本地缓存，不清空已展示内容

10. 非功能需求
================================================================================

  10.1 性能
      - 云函数方案生成耗时 ≤ 8s（含DeepSeek API调用）
      - 本地方案生成耗时 ≤ 300ms
      - 饮食生成耗时 ≤ 2s
      - 本周计划页首屏渲染 ≤ 500ms（本地缓存）

  10.2 安全
      - 用户身体数据加密存储
      - API Key 使用云函数环境变量，不硬编码
      - 医疗红线关键词拦截
      - 禁止明文传输隐私数据

  10.3 兼容性
      - 适配微信小程序（基础库 ≥ 2.2.3）
      - 适配所有手机尺寸（rpx响应式布局）

  10.4 可用性
      - 云端不可用时本地引擎兜底
      - 对话记忆双重存储（云端+本地）
      - 计划数据双重存储（云端+本地）
